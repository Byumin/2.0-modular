# Quality Score

## Purpose
이 문서는 이 저장소에서 작업 결과물을 어떤 기준으로 평가할지 정리한 공통 품질 평가 기준 문서다.

목적은 다음과 같다.

1. 결과물을 감각적으로만 판단하지 않고 기준표로 평가한다.
2. 무엇이 잘 됐고 무엇이 부족한지 항목별로 분리해서 본다.
3. 작업 실패 원인이 계획 문제인지, 실행 판단 문제인지 회고할 때 참고한다.
4. `docs/exec-plans/`에 남기는 실행 계획/회고 문서와 연결해 품질 판단 근거를 남긴다.

## When To Use
아래 작업이 들어갔으면 가능하면 이 기준으로 결과물을 평가한다.

- 기능 추가
- 버그 수정
- 리팩토링
- 구조 변경
- 프런트/UI 수정
- 문서 체계 정리
- API 변경

## UI Review Rule
UI나 프런트엔드 변경이 포함된 작업은 코드만 보고 평가하지 않는다.
반드시 실제 화면을 확인할 수 있는 스크린샷을 찍어서 레이아웃, 간격, 색상, 카드 크기, 영역 정렬, 텍스트 가독성, 반응형 붕괴 여부를 확인한 뒤 평가한다.

가능하면 아래 기준으로 확인한다.

- 주요 화면의 전체 페이지 스크린샷
- 변경된 컴포넌트가 보이는 근접 스크린샷
- 데스크톱 기준 확인
- 필요 시 모바일 또는 좁은 폭 화면 확인

스크린샷 기반 확인 없이 UI 품질을 높게 평가하지 않는다.

## Scoring Method
각 항목은 `1점 ~ 5점`으로 평가한다.

- `1점`: 매우 부족함
- `2점`: 부족함
- `3점`: 기본 수준
- `4점`: 좋음
- `5점`: 매우 좋음

총점은 모든 항목 합산 또는 필요한 항목만 선택해서 사용할 수 있다.

## Quality Categories
### 1. Requirement Fit
요청한 내용이 실제로 제대로 반영되었는지 평가한다.

- `1점`: 요구사항을 크게 벗어남
- `2점`: 핵심 요구사항 일부만 반영됨
- `3점`: 주요 요구사항은 반영됐지만 빠진 부분이 있음
- `4점`: 요구사항이 전반적으로 잘 반영됨
- `5점`: 요구사항이 정확하고 빠짐없이 반영됨

### 2. Functional Correctness
기능이 실제로 의도한 대로 동작하는지 평가한다.

- `1점`: 기능이 거의 동작하지 않음
- `2점`: 핵심 기능에 명확한 오류가 있음
- `3점`: 기본 동작은 하나 예외/분기 처리 부족
- `4점`: 기능이 안정적으로 동작함
- `5점`: 정상 동작과 예외 케이스까지 매우 잘 처리됨

### 3. Architectural Consistency
현재 저장소 구조와 계층 규칙에 맞게 구현되었는지 평가한다.

- `1점`: 구조를 심하게 깨뜨림
- `2점`: 계층 책임이 자주 섞임
- `3점`: 대체로 맞지만 일부 부조화가 있음
- `4점`: 기존 구조와 잘 맞음
- `5점`: 구조 일관성이 매우 높고 확장성도 좋음

### 4. Readability And Maintainability
코드와 문서가 읽기 쉽고 이후 수정하기 쉬운지 평가한다.

- `1점`: 읽기 어렵고 추적이 힘듦
- `2점`: 의미 파악이 자주 막힘
- `3점`: 이해 가능하지만 정리 여지가 큼
- `4점`: 명확하고 유지보수하기 좋음
- `5점`: 구조, 이름, 책임 분리가 매우 명확함

### 5. Validation And Testing
검증과 테스트가 충분했는지 평가한다.

- `1점`: 검증 없음
- `2점`: 최소 수준 확인만 함
- `3점`: 핵심 경로는 확인했음
- `4점`: 주요 흐름과 영향 범위를 충분히 검증함
- `5점`: 테스트/검증 근거가 매우 탄탄함

### 6. Edge Case Handling
예외 상황과 경계 조건을 고려했는지 평가한다.

- `1점`: 예외 케이스 고려 없음
- `2점`: 일부만 고려함
- `3점`: 자주 나오는 예외는 처리함
- `4점`: 주요 경계 조건을 잘 처리함
- `5점`: 예외/분기/실패 상황 대응이 매우 탄탄함

### 7. Documentation Quality
관련 문서가 충분하고 추후 참고 가능하게 정리되었는지 평가한다.

- `1점`: 문서 없음
- `2점`: 문서가 매우 불충분함
- `3점`: 기본 설명은 있음
- `4점`: 필요한 문서가 잘 정리됨
- `5점`: 문서 구조와 내용이 매우 명확하고 재사용 가능함

### 8. UI And Design Consistency
프런트 작업이 있을 때 화면 일관성과 디자인 규칙을 잘 지켰는지 평가한다.

- `1점`: 화면 일관성이 크게 깨짐
- `2점`: 스타일 차이가 큼
- `3점`: 기본은 맞지만 통일감이 약함
- `4점`: 기존 디자인 규칙과 잘 맞음
- `5점`: 화면 품질과 통일성이 매우 높음

평가 전 확인 항목:
- 실제 화면 스크린샷을 확인했는가
- 카드 크기, 패딩, 간격, 정렬선이 일관적인가
- 색상과 타이포그래피가 디자인 가이드와 맞는가
- 깨짐이나 과도한 밀집이 없는가
- 필요 시 반응형 화면도 확인했는가

### 9. Regression Risk
기존 기능을 깨뜨릴 위험이 낮은지 평가한다.

- `1점`: 회귀 위험이 매우 큼
- `2점`: 위험 요소가 뚜렷함
- `3점`: 위험이 관리 가능 수준
- `4점`: 회귀 가능성이 낮음
- `5점`: 영향 범위를 잘 통제했고 위험이 매우 낮음

### 10. Completion Level
이번 작업이 실제로 마무리 가능한 수준까지 도달했는지 평가한다.

- `1점`: 거의 미완료
- `2점`: 절반 이하 수준
- `3점`: 핵심은 끝났지만 후속 작업이 큼
- `4점`: 실사용 가능한 수준으로 마무리됨
- `5점`: 후속 조치까지 고려된 높은 완료도

## Score Interpretation
총점 또는 평균 점수는 아래처럼 해석한다.

- `4.5 ~ 5.0`: 매우 좋음
- `3.8 ~ 4.4`: 충분히 좋음
- `3.0 ~ 3.7`: 기본 수준
- `2.0 ~ 2.9`: 보완 필요
- `1.0 ~ 1.9`: 재작업 필요

## Minimum Mandatory Check
아래 항목은 점수와 별개로 반드시 확인하는 것이 좋다.

- 요청한 기능이 실제로 반영되었는가
- 기존 엔트리포인트와 계층 구조를 잘못 해석하지 않았는가
- 영향 범위가 큰 변경이면 검증 근거가 있는가
- 문서가 필요한 작업이면 관련 문서가 갱신되었는가
- 실패가 있었다면 원인을 회고 문서에 남겼는가
- UI/프런트 변경이 있었다면 스크린샷으로 실제 화면을 확인했는가

## Link With Execution Plans
실행 계획 문서와 이 문서는 함께 사용하는 것을 권장한다.

권장 방식:

1. `docs/exec-plans/`에 작업 계획 문서를 만든다.
2. 작업 완료 후 이 문서 기준으로 품질을 항목별 평가한다.
3. 점수가 낮은 항목은 회고 섹션에서 원인을 적는다.
4. 그 원인이 `Plan Problem`인지 `Execution Judgment Problem`인지 분류한다.

## Example Format
아래 형식으로 각 작업 회고 문서에 붙여서 사용할 수 있다.

```md
## Quality Score
- Requirement Fit: 5/5
- Functional Correctness: 4/5
- Architectural Consistency: 4/5
- Readability And Maintainability: 4/5
- Validation And Testing: 3/5
- Edge Case Handling: 3/5
- Documentation Quality: 5/5
- Regression Risk: 4/5
- Completion Level: 4/5

### Summary
- Average: 4.0
- Notes: 테스트 근거가 약해서 Validation 점수를 낮게 평가함.
```

## Rule Of Use
이 문서는 절대적인 정답표가 아니라, 작업 결과를 더 분명하게 판단하기 위한 공통 기준표다.
중요한 것은 숫자 자체보다 왜 그 점수를 줬는지 근거를 남기는 것이다.
