# 내담자 수집 Phase 1 상세 설계 문서화

## 요청 요약
- `Client Intake Policy`를 바탕으로 Phase 1 범위의 DB/스키마/API 변경안을 문서화한다.
- 승인 모드는 제외하고 `pre_registered_only`와 `auto_create`만 설계한다.
- 실제 DB 수정이나 마이그레이션은 하지 않고, 승인 게이트만 문서에 명시한다.

## 작업 목표
- Phase 1 구현에 필요한 상세 설계 문서를 만든다.
- DB 변경안, 스키마 변경안, API 변경안, 서비스 분기안을 한 문서에 정리한다.
- DB 관련 실제 작업은 사용자 승인 후 진행한다는 경계를 분명히 남긴다.

## 초기 가설
- 정책 문서와 상세 변경안 문서는 분리해야 한다.
- 상세 변경안은 `docs/features/` 아래 별도 문서로 두고, 정책 문서와 허브에는 링크만 추가하는 것이 유지보수에 적절하다.

## 실행 계획
1. 현재 커스텀 검사/평가 링크/내담자 관련 스키마와 API 범위를 확인한다.
2. Phase 1 전용 상세 설계 문서를 작성한다.
3. 문서에 DB 승인 게이트와 미합의 설계 포인트를 분리한다.
4. 정책 문서와 기능 허브에 새 문서 링크만 추가한다.

## 작업 중 변경 사항
- 초기 문서 초안에서는 `validate-profile` 통합안과 `register-client` 유지안을 함께 비교하도록 정리했다.
- DB 변경은 허용된 작업이 아니므로 실제 구현 단계와 분리해 `Approval Checklist` 섹션을 별도로 뒀다.
- 이후 합의에 따라 `register-client` API는 유지하는 쪽으로 확정했다.
- 생년월일이 없는 유사 매칭은 Phase 1 exact match 재사용 대상에서 제외하고, Phase 2 관리자 승인 흐름으로 넘기도록 문서를 보강했다.
- 애매 매칭 승인 흐름은 별도 상세 문서로 분리하는 편이 맞다고 판단해 `client-intake-phase2-ambiguous-match-spec.md`를 추가했다.

## 결과
- `docs/features/client-intake-phase1-spec.md`를 작성했다.
- 문서에 DB, 스키마, API, 서비스 로직, 승인 게이트, 미합의 설계 포인트를 포함했다.
- 유사 매칭 승인 흐름을 후속 단계로 분리해 정책 문서와 상세 설계 문서에 함께 반영했다.
- Phase 2용 `애매 매칭 승인` 상세 설계 문서를 별도로 추가했다.

## 검증 내용
- 상세 설계 문서가 정책 문서와 중복되지 않고 구현 관점으로 한 단계 더 내려왔는지 확인한다.
- DB 관련 실제 수정 없이 설계와 승인 포인트만 정리했는지 확인한다.

## 회고
- 정책 문서와 상세 설계 문서를 분리한 덕분에 “왜 이 정책을 쓰는가”와 “어떻게 구현할 것인가”가 섞이지 않았다.
- 승인 게이트를 명시해두면 이후 DB 작업 직전에 다시 범위를 합의하기 쉽다.
- Phase 1과 Phase 2 경계를 명확히 나누니, 자동 생성의 운영 효율과 데이터 품질 문제를 같은 턴에 과도하게 풀려고 하지 않아도 됐다.
