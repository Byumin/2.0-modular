# 내담자 자동 배정 정책 문서화

## 요청 요약
- 내담자 자동 배정 방식과 사전 등록 방식, 혼합 운영 방식을 서비스 관점에서 먼저 문서화한다.
- 이후 개발이 이 문서를 기준으로 진행될 수 있게 기능 정책, 장단점, 운영 모드, 단계별 로드맵을 정리한다.

## 작업 목표
- 내담자 생성/배정 정책의 source of truth 문서를 만든다.
- 기존 기능 문서와 중복되지 않도록 적절한 문서 위치를 선택한다.
- 기능 허브에서 새 정책 문서로 진입할 수 있게 연결한다.

## 초기 가설
- 이 주제는 `assessment-link-flow`와 `client-management` 양쪽에 걸치지만, 링크 진입 정책과 자동 생성/배정 분기가 핵심이므로 기능 문서군 아래 별도 정책 문서로 분리하는 편이 적절하다.
- 기존 기능 문서에는 전체 정책 본문을 복제하지 않고 링크만 추가하는 것이 유지보수에 유리하다.

## 실행 계획
1. 기존 기능 문서와 문서 거버넌스 규칙을 확인한다.
2. 새 정책 문서를 `docs/features/` 아래에 작성한다.
3. 문서에는 운영 모드, 장단점, 추천 방향, 중복 내담자 대응, 단계별 개발 로드맵을 포함한다.
4. `docs/features/README.md`와 관련 기능 문서에는 요약 링크만 추가한다.

## 작업 중 변경 사항
- 정책 본문을 기존 `assessment-link-flow.md`에 직접 확장하지 않고, `docs/features/client-intake-policy.md`로 분리했다.
- 기존 기능 문서들에는 전체 정책을 복제하지 않고 source of truth 링크만 추가했다.

## 결과
- `docs/features/client-intake-policy.md`를 새로 작성했다.
- `docs/features/README.md`에 새 문서를 추가했다.
- `docs/features/assessment-link-flow.md`와 `docs/features/client-management.md`에는 정책 source 링크만 추가했다.

## 검증 내용
- 문서 역할과 source of truth 위치가 `docs/doc-governance.md` 기준과 맞는지 확인한다.
- 기능 허브에서 새 문서로 진입 가능한지 확인한다.
- 정책 본문이 기존 기능 문서에 중복 복제되지 않았는지 확인한다.

## 회고
- 분리 판단은 적절했다. 이 주제는 링크 흐름 설명을 넘어서 서비스 운영 정책과 로드맵까지 포함하므로, 기존 기능 문서에 덧붙이면 목적이 흐려졌을 가능성이 크다.
- source 문서를 하나로 두고 기존 기능 문서에는 링크만 둔 구조가 이후 정책 변경에도 가장 유지보수성이 높다.
