# Execution Plans

이 폴더는 작업 전 실행 계획과 작업 후 회고를 함께 남기는 문서를 보관하는 경로다.

## Why
실제 작업에서는 아래 두 경우를 나중에 분리해서 봐야 한다.

1. 처음 세운 계획 자체가 틀린 경우
2. 계획은 괜찮았지만 작업 중 판단이나 구현이 잘못된 경우

이 둘은 원인이 다르기 때문에, 매 작업마다 실행 계획 문서를 남겨야 이후 회고와 개선이 가능하다.

## Rule
- 실제 수정이 들어가는 작업은 가능한 한 먼저 실행 계획 문서를 만든다.
- 문서는 작업 시작 시 1차 작성하고, 작업 중간에 반드시 갱신한다.
- 작업 도중 계획이 바뀌면 바뀐 사실과 이유를 기록한다.
- 최종적으로 결과가 기대와 다르면 원인을 반드시 분류한다.

## Retrospective Classification
회고 시 아래처럼 분류한다.

### 1. Plan Problem
처음 세운 접근 자체가 틀렸거나, 조사 범위가 부족했거나, 잘못된 가정 위에서 출발한 경우다.

예시:
- 실제 엔트리포인트를 잘못 잡음
- 수정해야 할 계층을 잘못 선택함
- 요구사항 해석이 잘못되어 다른 방향으로 구현함

### 2. Execution Judgment Problem
계획 방향은 맞았지만 작업 중 세부 판단, 구현, 분기 처리, 검증이 잘못된 경우다.

예시:
- 중간 예외 케이스를 놓침
- 기존 함수 호출 관계를 잘못 읽음
- 테스트/검증 부족으로 잘못된 결과를 남김

### 3. Mixed
계획 문제와 실행 판단 문제가 동시에 있는 경우다.

## Naming Rule
파일명은 날짜와 작업 주제가 드러나게 작성한다.

예시:
- `2026-04-08-admin-login-flow-fix.md`
- `2026-04-08-custom-test-create-refactor.md`
- `2026-04-08-dashboard-ui-cleanup.md`

## Minimum Sections
각 실행 계획 문서는 최소 아래 섹션을 포함한다.

1. 작업 제목
2. 요청 요약
3. 작업 목표
4. 초기 가설
5. 실행 계획
6. 작업 중 변경 사항
7. 결과
8. 검증 내용
9. 회고

## Template
새 문서를 만들 때는 `_template.md`를 기준으로 복사해서 사용한다.
