# 검사 링크 미배정 내담자 등록 모달

## 작업 목표
- 검사 링크에서 배정되지 않은 내담자가 인적사항을 입력하고 진행하려 할 때, 인적사항 화면에서 확인 모달을 띄운다.
- 사용자가 동의하면 해당 관리자 소속 내담자로 등록하고 현재 검사에 자동 배정한 뒤 검사를 계속 진행시킨다.
- 사용자가 거절하면 검사를 취소하고 인적사항 단계에 머무르게 한다.

## 초기 가설과 접근 방식
- 현재 공개 링크 흐름은 `validate-profile`과 `submit` 모두 `find_assigned_client_for_profile()`에서 배정 여부를 확인한다.
- 프런트는 `403`을 모두 일반 오류로 처리하므로, `배정 없음` 케이스만 식별할 수 있도록 구조화된 에러 코드가 필요하다.
- 공개 링크에서 관리자 세션 없이 처리해야 하므로, 별도의 공개 API에서 `access_token -> admin_user_id/custom_test_id`를 해석해 내담자 생성과 배정을 수행해야 한다.
- 기존 관리자 내담자 생성 로직과 최대한 같은 정규화 규칙을 재사용하되, 중복 내담자 생성을 줄이기 위해 동일 관리자 아래 같은 인적사항의 기존 내담자가 있으면 우선 재사용한다.

## 단계별 실행 계획
1. 인적사항 화면 HTML/CSS/JS 구조와 기존 모달 스타일을 확인한다.
2. 공개 링크용 내담자 자동 등록/배정 서비스와 라우터, 요청 스키마를 추가한다.
3. `배정 없음` 오류를 프런트가 구분할 수 있도록 에러 코드를 정리한다.
4. 인적사항 화면에 확인 모달을 추가하고, `예/아니요` 흐름을 연결한다.
5. 수정 전/후 스크린샷을 캡처해 실제 동작과 레이아웃을 확인한다.

## 작업 중 변경된 계획과 변경 이유
- 실동작 검증은 운영 DB를 직접 건드리지 않기 위해 `modular.db` 복사본을 `/tmp/modular-unassigned-test.db`로 만들어 서비스 함수를 직접 호출하는 방식으로 진행했다.
- UI 캡처는 앱 서버를 실제로 띄워 확인하려 했지만, 현재 로컬 환경에서 `FastAPI`와 `Starlette` 버전 충돌로 `APIRouter()` 초기화 시 `Router.__init__() got an unexpected keyword argument 'on_startup'`가 발생했다. 그래서 수정 전 화면은 정적 페이지 기준으로 캡처하고, 수정 후 모달 배치는 동일 CSS/마크업을 사용한 로컬 preview HTML로 확인했다.

## 최종 결과
- 공개 검사 링크 흐름에 `POST /api/assessment-links/{access_token}/register-client`를 추가했다.
- `validate-profile`이 `배정 없음` 케이스에서 구조화된 오류 코드 `UNASSIGNED_CLIENT_REQUIRED`를 반환하도록 정리했다.
- 인적사항 화면에 `배정되지 않은 내담자` 확인 모달과 `예/아니요` 흐름을 추가했다.
- `예`를 누르면 내담자를 자동 등록하거나 기존 동일 인적사항 내담자를 재사용해 현재 검사에 배정한 뒤 같은 프로필로 다시 검증을 시도한다.
- `아니요`를 누르면 등록 없이 인적사항 단계에 남는다.
- 자동 등록 API는 현재 검사에 이미 배정된 내담자가 하나라도 있으면 동작하지 않도록 막아 공개 링크 우회를 방지했다.
- 검증 스크립트로 `배정 없음 -> 등록/배정 -> 재검증 통과` 흐름을 확인했다.
