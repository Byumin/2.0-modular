# Assessment Session Navigation UX Questions

## 문서 역할
- 역할: 다중 세션 검사 실시 UX 정책 결정을 위한 질문/제안 문서.
- 독자: 제품 의사결정자, 프론트엔드 구현자, 검사 운영자.
- 상태: 미결정 정책 정리 문서이며, 현재 구현의 source-of-truth가 아니다.
- 연결 문서: `docs/features/assessment-link-flow.md`, `frontend/src/pages/assessment/AssessmentPage.tsx`, `frontend/src/pages/assessment/steps/QuestionStep.tsx`.

## 현재 구현 요약
- 세션은 `AssessmentPage`의 `activeSessionIndex`로 관리된다.
- 세션 1을 제출하면 `handleSessionSubmit`이 `activeSessionIndex`를 증가시키고 다음 세션 안내 화면으로 이동한다.
- `QuestionStep`에는 현재 세션의 `activeSessionParts`만 전달된다.
- 따라서 세션 2 문항 화면에서는 세션 1 문항이 렌더링되지 않으며, 세션 1로 돌아가는 UI도 없다.
- `QuestionStep` 내부의 이전/다음/파트 이동은 현재 전달된 parts 안에서만 동작한다. 즉 세션 간 이동이 아니라 같은 세션 내부 이동이다.

## 핵심 질문
1. 세션이 여러 개일 때, 다음 세션으로 넘어간 뒤 이전 세션으로 돌아가지 못하게 하는 것이 맞는가?
2. 이전 세션으로 돌아갈 수 있게 한다면, 응답 수정과 임시저장/최종 제출 데이터 병합은 어떻게 보장할 것인가?
3. 세션 목록 이동을 제공한다면, 세션 1을 실시하지 않고 세션 2로 뛰어넘는 것을 허용할 것인가?
4. 뛰어넘기를 막는다면, 완료된 세션과 미완료 세션을 UI에서 어떻게 구분할 것인가?
5. 이전 세션으로 못 돌아가게 한다면, 세션 제출 전 모달에 "다음 세션으로 이동하면 이전 세션으로 돌아올 수 없습니다" 같은 안내를 반드시 넣을 것인가?
6. 미응답 제출 허용 검사에서 세션 1에 미응답을 남긴 채 세션 2로 넘어가는 것이 허용되는가?
7. 미응답 제출을 허용하더라도, 세션 전환 시에는 최종 제출보다 더 강한 경고가 필요한가?
8. 세션 2 이후 브라우저 새로고침/재접속 시 어느 세션부터 재개해야 하는가?
9. 운영자가 특정 검사에서 "세션 이동 가능" 또는 "순차 잠금"을 선택할 수 있어야 하는가?
10. 세션 안내 화면에서 이전 세션의 완료 상태, 문항 수, 미응답 수를 보여줘야 하는가?

## 정책 선택지

### 선택지 A: 순차 잠금
세션 1을 제출해 세션 2로 넘어가면 세션 1로 돌아갈 수 없다.

장점:
- 검사 흐름이 단순하다.
- 세션별 시간 제한, 안내문, 조건이 다른 검사에서 운영 리스크가 낮다.
- 수검자가 이전 응답을 보고 이후 세션 응답을 바꾸는 문제를 줄일 수 있다.

단점:
- 실수로 다음 세션으로 넘어간 경우 수정할 수 없다.
- 세션 전환 전 경고/확인 UX가 없으면 불친절하게 느껴질 수 있다.
- 관리자 문의가 늘 수 있다.

필요 UI:
- 세션 제출 버튼 클릭 시 확인 모달.
- 모달 문구 예시:
  - 제목: `세션을 제출하고 다음으로 이동할까요?`
  - 본문: `다음 세션으로 이동하면 현재 세션으로 돌아와 응답을 수정할 수 없습니다.`
  - 보조 정보: `미응답 문항이 있는 경우 그대로 저장됩니다.`
  - 버튼: `계속 응답하기`, `다음 세션으로 이동`

권장 조건:
- 세션별 검사 조건이 다르거나, 순서 효과를 통제해야 하는 검사는 이 방식을 기본값으로 둔다.

### 선택지 B: 완료 세션 복귀 허용
세션 2에서도 세션 1로 돌아가 응답을 수정할 수 있다.

장점:
- 수검자가 실수한 응답을 직접 수정할 수 있다.
- 일반 설문처럼 편안한 탐색 경험을 제공한다.
- 제출 전 전체 검토 UX를 만들기 쉽다.

단점:
- 세션 순서 통제가 약해진다.
- 세션별 안내/조건이 다른 경우 수검자가 맥락을 섞어서 응답할 수 있다.
- 임시저장, 미응답 상태, 세션별 완료 상태 UI가 더 복잡해진다.

필요 UI:
- 상단 또는 사이드바에 세션 탭/목록.
- 세션 상태 표시:
  - `진행 중`
  - `완료`
  - `미응답 n개`
  - `잠김`
- 완료 세션 복귀 시 안내:
  - `이전 세션 응답을 수정하면 전체 제출 내용이 변경됩니다.`

권장 조건:
- 세션이 검사 통제 단위가 아니라 화면 구성 단위에 가까운 경우에만 허용한다.

### 선택지 C: 순차 진행 + 최종 검토 단계
세션 진행 중에는 이전 세션으로 돌아갈 수 없지만, 마지막 최종 제출 전 전체 검토 단계에서 세션별 미응답/수정 진입을 허용한다.

장점:
- 진행 중에는 순서를 유지한다.
- 최종 제출 전 실수 수정 기회를 제공한다.
- 운영자와 수검자 모두에게 납득 가능한 균형점이다.

단점:
- 최종 검토 화면을 별도로 구현해야 한다.
- 세션별 응답 수정 후 다시 검토 화면으로 돌아오는 흐름이 필요하다.

필요 UI:
- 마지막 세션 완료 후 `최종 검토` 화면.
- 세션별 요약:
  - `세션 1: 완료 / 미응답 0개`
  - `세션 2: 미응답 3개`
- 수정 버튼:
  - `세션 1 수정`
  - `세션 2 수정`
- 최종 제출 버튼.

권장 조건:
- 검사 신뢰도를 위해 순차 진행은 유지하되, 사용자 실수에 대한 안전장치가 필요한 경우.

### 선택지 D: 관리자 설정 기반
검사 생성/수정 단계에서 세션 이동 정책을 선택한다.

설정 예시:
- `순차 잠금`: 다음 세션 진입 후 이전 세션 수정 불가.
- `완료 세션 복귀 허용`: 이전 세션 수정 가능.
- `최종 검토에서만 수정 허용`: 진행 중 복귀 불가, 최종 검토에서 수정 가능.

장점:
- 검사 성격별로 다른 정책을 적용할 수 있다.
- 운영자가 통제 수준을 직접 선택할 수 있다.

단점:
- 관리자 UI와 저장 스키마가 필요하다.
- 수검자 UI 분기가 늘어난다.
- 설정 설명을 잘못 이해하면 운영 오류가 생길 수 있다.

권장 조건:
- 검사 종류가 다양하고, 어떤 검사는 순차 통제가 필요하며 어떤 검사는 자유 이동이 필요한 경우.

## 미응답과 세션 전환 모달

### 미응답 제출 비허용 검사
- 세션 내 모든 문항에 응답해야 다음 세션으로 이동할 수 있게 한다.
- 이 경우 세션 전환 모달은 단순 확인만 하면 된다.
- 문구 예시:
  - `현재 세션 응답이 완료되었습니다. 다음 세션으로 이동하면 이전 세션으로 돌아올 수 없습니다.`

### 미응답 제출 허용 검사
- 세션 전환 시 미응답 문항이 남아 있을 수 있다.
- 이 경우 일반 제출보다 강한 확인이 필요하다.
- 문구 예시:
  - 제목: `미응답 문항이 있습니다`
  - 본문: `현재 세션에 총 n개 문항이 미응답 상태입니다. 다음 세션으로 이동하면 현재 세션으로 돌아와 응답을 수정할 수 없습니다.`
  - 버튼: `첫 미응답 이동`, `미응답으로 두고 다음 세션 이동`

## 뛰어넘기 정책

### 세션 뛰어넘기 비허용
- 세션 1을 완료하지 않으면 세션 2 시작 불가.
- 가장 안전한 기본값이다.
- 세션 목록이 있더라도 미래 세션은 `잠김`으로 표시한다.

### 세션 뛰어넘기 허용
- 세션을 설문 섹션처럼 취급할 때만 적합하다.
- 세션별 완료/미완료 상태가 명확해야 한다.
- 최종 제출 전 전체 미응답/미완료 검토가 필수다.

권장:
- 기본값은 `뛰어넘기 비허용`.
- 뛰어넘기 허용은 별도 관리자 설정으로만 제공한다.

## 권장 기본안
현재 검사 실시 흐름에는 `선택지 A: 순차 잠금`을 기본으로 유지하는 것이 가장 보수적이다.

다만 순차 잠금을 유지한다면 다음 보완이 필요하다.
- 세션 전환 버튼 클릭 시 확인 모달을 추가한다.
- 모달에 `이동 후 이전 세션으로 돌아올 수 없음`을 명시한다.
- 미응답이 있는 경우 미응답 개수와 되돌아갈 수 없다는 사실을 같은 모달에서 함께 안내한다.
- 세션 안내 화면에는 현재 세션 번호와 전체 세션 수를 명확히 표시한다.

중장기적으로는 `선택지 C: 순차 진행 + 최종 검토 단계`가 가장 균형 잡힌 UX다.

## 구현 시 체크리스트
- `AssessmentPage`에서 세션 정책 상태를 명확히 둔다.
- `QuestionStep`은 현재 세션 내부 이동만 책임지도록 유지한다.
- 세션 간 이동은 `AssessmentPage`에서만 처리한다.
- 세션 제출 전 모달은 `QuestionStep` 내부 미응답 모달과 충돌하지 않게 설계한다.
- 임시저장 payload의 `current_part_index`, `current_page`가 현재 세션 기준인지 전체 검사 기준인지 문서화한다.
- 최종 제출 payload에는 모든 세션 응답이 누락 없이 병합되어야 한다.
- 세션 복귀 허용 정책을 도입할 경우 `activeSessionIndex`, `sessionAnswers`, `activeDraft` 복원 규칙을 함께 바꾼다.

## 결정 필요 항목
1. 기본 정책을 `순차 잠금`으로 확정할지.
2. 세션 전환 확인 모달을 즉시 추가할지.
3. 미응답 허용 검사에서 세션 전환 모달 문구를 별도로 둘지.
4. 최종 검토 단계를 이번 범위에 포함할지, 후속 과제로 둘지.
5. 관리자 설정으로 세션 이동 정책을 노출할지.

## 관련 코드
- `frontend/src/pages/assessment/AssessmentPage.tsx`: 세션 인덱스, 세션 전환, 전체 응답 병합.
- `frontend/src/pages/assessment/steps/QuestionStep.tsx`: 현재 세션 내부 part/page 이동, 미응답 확인 모달.
- `frontend/src/pages/assessment/steps/IntroStep.tsx`: 세션 안내 화면.
